**Q4:** FC69 is about a factory validating env before applying config. FC10 already covers the Fernet singleton pattern (cached at import time, RuntimeError outside app context). Are these genuinely different?

**A4:** Yes. FC10 covers fail-open behavior -- infrastructure errors that silently grant access or suppress exceptions. FC69 is about an ordering trap in the factory initialization sequence: a secret is validated FROM ENV before the caller's config dict is merged, making the config-dict path for that secret silently ineffective. The distinction is: FC10 = wrong error handling (fail-open), FC69 = wrong initialization order (caller's config can't override what was already validated). An agent could correctly implement FC10's "never fail-open" rule while still hitting FC69 if it builds a factory that validates before merging config. They are orthogonal agent rules.
