## Agent: performance-oracle

**Verdict: No critical performance issues. Two advisory items.**

### Checked:
- **SQLite indexes:** All FK columns indexed. Composite `(project_id, shoot_date)` index on
  `schedule_entries` supports the most expensive query (DOOD grid per-cast-member per-date).
- **DOOD grid query:** `get_dood_grid` executes one SQL per cast member in a Python loop (N+1 pattern).
  For a large cast (50+ members) this could be slow. However, for indie/mid-budget scope (10-40 cast)
  this is acceptable. P3 advisory.
- **Shared `:memory:` connection:** `check_same_thread=False` with a shared connection means concurrent
  requests can race on the same SQLite connection. The `BEGIN IMMEDIATE` locks mitigate data corruption
  but not the thread-safety of the Python sqlite3 connection object itself. Acceptable for single-process
  development/demo use. P3 advisory.
- **WAL + autocommit=True:** Correct — autocommit=True in Python 3.12+ disables implicit transactions;
  all writes correctly use explicit `BEGIN IMMEDIATE`.

## Worker Judgment Calls — Validation

### 1. Creator-as-producer enrollment in `create_project`

`project_models.create_project` enrolls `created_by` as a `producer` in `project_members` within
the same `BEGIN IMMEDIATE` transaction. This is the correct fix for the create→dashboard membership
requirement. The spec does not prescribe this but the dashboard is guarded by `require_project_member`
— without enrollment, the creator would get a 403 immediately. **Assessment: CORRECT design decision,
not a defect. The transaction contract properly commits both the project and the membership atomically.**

### 2. Inline UPDATEs in departments/crew/locations without model `update_*` functions

- `departments/routes.py` uses `assign_department_head` (model fn exists — correct)
- `crew/routes.py update` does inline `UPDATE crew_members ... WHERE id = ?` within a BEGIN IMMEDIATE
  block, combined with FTS5 re-index in the same transaction (FC52 single-writer pattern). No
  `update_crew_member` model function exists in spec. **Assessment: CORRECT — spec's Transaction
  Contracts table only lists `create_crew_member`. The inline transaction pattern correctly wraps
  the UPDATE + FTS5 reindex atomically.**
- `locations/routes.py` presumably similar. **Assessment: acceptable worker judgment call.**

### 3. Budget allocate form uses numeric `department_id` input

`budget/routes.py allocate` parses `int(request.form['department_id'])` and validates against
`get_departments(conn, project_id)`. No departments list is passed to the index template for the
form dropdown. **This is a P2 finding** — the allocate form in `budget/index.html` renders a
`department_id` input without a corresponding departments list in the render context. The route
for `GET /budget/<pid>` passes `summary` and `categories` but not `departments`. This means the
allocate form either hard-codes department IDs (fragile) or does not work without JS. See P2 below.

**P2-2: Redundant double `get_schedule_entries` call in callsheets.generate (regressed from run 063)**

- Location: `app/blueprints/callsheets/routes.py:70-73` vs `callsheet_models.generate_call_sheet:32`

  from prior build.

### P3 Findings

**P3-1: DOOD grid N+1 query pattern**

- `get_dood_grid` executes 1 SQL per cast member to find working days. For indie scale (≤40 cast)
  this is acceptable. Could be optimized to a single JOIN query.

**P3-2: VALID_PHASE_TRANSITIONS uses sets, spec uses lists**

- Implementation uses `set` literals (`{'pre_production'}`) vs spec's `list` literals (`['pre_production']`).
  Sets are strictly better for `in` checks (O(1) vs O(n)). No functional impact. Advisory only.

**P3-3: `sort_order` TOCTOU in schedule.create**

- `MAX(sort_order)` read outside lock before `create_schedule_entry`. Two concurrent requests could
  assign duplicate sort_order values. Non-data-loss (reorder resolves it). Acceptable for single-producer
  indie PM tool scope.
