**PROPOSED gap decisions (2026-06-11 — NOT human-frozen; operator should review, but the spec implements these verbatim unless overridden):**
1. **Scoring model** — every scoring signal lives on answer options (`option_scores`: option → pain category → points ≥ 0). Per category: `raw_points` = sum of selected options' points; `max_points` = sum over active questions of that question's maximum contribution (single_choice: highest option score; multi_choice: sum of ALL option scores, since all are selectable); `severity_pct = (raw_points * 100) // max_points` (integer floor — no banker's-rounding flake). Bands derived, never stored: low 0–24, moderate 25–49, high 50–74, critical 75–100. *Reasoning: option-level weights keep the engine a pure table-driven function that swarm agents can build and test deterministically.*
2. **Prescription rule semantics** — "reported categories" = categories with severity ≥ 25, ordered severity desc (ties: raw_points desc, then category key A→Z), capped at 5. For each reported category in that order, EVERY active rule matching (`rule.pain_category == category` AND `rule.min_severity_pct <= severity_pct` AND ordinal constraint filters pass AND its solution is not archived) fires, in `priority` asc / `id` asc order; duplicate solutions are dropped first-wins via `INSERT OR IGNORE` on `UNIQUE(prescription_id, solution_id)`. Constraint filters are ordinal minimums (`budget_min`, `maturity_min`; NULL = no filter) against the derived constraint profile. *Reasoning: all-matching-rules (not top-rule-only) naturally fills the Quick Win / Core / Growth tiers, which is the sales narrative.*
3. **Branching model** — linear question sequence by global unique `sort_order` with single-level conditionals: a question may declare `depends_on_qkey` + `depends_on_value`; it is visible only when that single_choice answer matches. No graph, no multi-condition logic, no chained dependencies (a dependent question may not itself be depended on). *Reasoning: deterministic, testable, and enough adaptivity for a 25-question sales interview.*
4. **Token access + lifecycle** — clients never get accounts. Engagements carry `interview_token` (and after publish, `report_token`), both `secrets.token_urlsafe(32)`, looked up exactly; unknown token → 404. Status machine: `created → in_progress` (first answer) `→ completed` (last answer; diagnosis + prescription generated in the same transaction) `→ published` (consultant publishes report). Recompute allowed in completed/published and WIPES consultant overrides (UI must warn). *Reasoning: tokens make the demo frictionless for prospects; a tiny status machine keeps the Authorization Matrix small.*
5. **Seed determinism** — the spec pins the full seed verbatim: 26 questions (4 free-text, 1 multi-choice, 1 dependent) across 6 sections, 8 pain categories, 18 solutions (explicit IDs 1–18), 23 rules (explicit IDs 1–23), one demo engagement ("Sunset Yoga Studio") with fixed tokens `demo-interview-token-0001` / `demo-report-token-0001`, a pinned answer set, and hand-computed expected severities (scheduling 100 critical, manual 83 critical, visibility 72 high, billing 70 high, disconnected 64 high) so smoke tests and the scoring critical-flow test assert exact numbers. *Reasoning: the demo engagement makes the app sales-ready at first boot and gives the swarm a deterministic oracle.*

**Other PROPOSED decisions:** app name **StackRx** (prescription metaphor; kebab `stack-rx`); house stack Flask 3 + SQLite + Jinja2 + Bootstrap 5 dark + print CSS (matches every prior swarm build — zero new infra risk); **rule-based engine, NO LLM and NO external API calls in v1** (the demo must never depend on an API key, network latency, or model behavior in a meeting; UI styling loads Bootstrap from the jsdelivr CDN — vendoring assets for fully-offline demos is a deferred option; an LLM narrative-polish layer is explicitly deferred); 11-agent swarm (scaffold, database, auth, engagements, interview, questions, diagnosis, prescription, catalog, reports, tests); single consultant user seeded from env — `ADMIN_PASSWORD` required (fail-closed), `ADMIN_USERNAME` defaults to `consultant`, no register route; free-text answers required non-empty (they power the report's "in their own words" quotes); no client back-editing of answers in v1 (deferred); constraint profile derived from answers at read time, never stored (single source of truth).
**8. Tone:** Hyper-precise, literal, strictly typed, table-driven; client-facing surfaces polished and confident — this is a sales artifact, not an internal tool.
**9. Avoid:** Do NOT call any LLM or external API (the engine is rule-based; this is a declared constraint, not an omission). Do NOT touch any production DB (local SQLite only). Do NOT store derived values (bands, constraint profiles, remaining budgets). Do NOT let clients reach any consultant route or any other engagement's data (token scoping is absolute). Do NOT allow overlapping file ownership across the 11 agents (overlap = gate fail). Do NOT implement client accounts, multi-consultant tenancy, answer back-editing, email sending, or LLM polish (all deferred).

## Feed-Forward
- **Hardest decision:** Rule-based deterministic engine vs. LLM-driven interview. Rule-based won because a sales demo must be flawless with zero API dependency and zero latency, and the operating contract forbids undeclared external calls — but it trades away conversational flexibility.
- **Rejected alternatives:** LLM interview/prescription (reliability + contract risk); client accounts (friction kills prospect completion); storing computed bands/profiles (derived-value drift, prior-build P1 class); graph-based branching (untestable at swarm scale).
- **Least confident:** Internal consistency of the seed data web — 26 questions × option scores × 8 categories × 23 rules × 18 solutions × hand-computed demo severities. A single wrong points value silently shifts a demo severity and breaks the pinned acceptance numbers. This is exactly what the adversarial verifier passes and critical-flow test 3 must scrutinize.
